# 工程开发规范

## 1. 目录规范

### 1.1 顶层目录职责

- `include/`
  公共头文件，对外暴露的接口和类型定义
- `src/`
  项目核心实现，按模块拆目录
- `examples/`
  最小可运行示例，不承载核心框架逻辑
- `tests/`
  自动化测试
- `docs/`
  规划文档、开发记录、规范说明
- `cmake/`
  CMake 依赖和公共脚本

### 1.2 模块划分规则

- `app`
  日志、配置、基础设施
- `net`
  网络连接、协议、分发器、限流、定时器
- `game/gateway`
  网关接入与路由
- `game/login`
  登录认证
- `game/room`
  房间逻辑
- `game/battle`
  战斗逻辑

## 2. 代码组织规范

### 2.1 头文件与实现文件

- `.h` 只放接口、类型声明、必要的 inline 代码
- `.cpp` 放具体实现
- 避免在头文件里写大段复杂逻辑

### 2.2 模块依赖方向

依赖方向必须尽量单向：

```text
app -> net -> game
```

约束：

- `net` 不能依赖 `game`
- `game/login`、`game/room`、`game/battle` 不应该相互直接强耦合
- 跨模块协作优先通过接口、消息或统一上下文对象

### 2.3 示例代码规则

- `examples/` 只用于展示框架能力
- 不要把业务核心实现放在 `examples/` 里
- 示例逻辑要尽量薄，只负责组合已有模块

## 3. CMake 规范

### 3.1 根 CMake 职责

根 `CMakeLists.txt` 只负责：

- 工程全局设置
- 公共依赖加载
- 子目录挂接

不要在根 CMake 里堆所有目标定义。

### 3.2 子 CMake 职责

- `src/CMakeLists.txt`：定义核心库目标
- `src/<module>/CMakeLists.txt`：向对应目标追加源文件
- `examples/<name>/CMakeLists.txt`：定义示例可执行文件
- `tests/<scope>/CMakeLists.txt`：定义测试目标

### 3.3 目标命名建议

- 核心库：`project_app`、`project_net`、`project_game`
- 示例程序：`example_xxx` 或具名程序
- 测试目标：`project_unit_tests`、`project_integration_tests`

## 4. 测试规范

### 4.1 单元测试

要求：

- 不依赖真实网络环境
- 尽量小而快
- 覆盖协议、纯逻辑、边界值

### 4.2 集成测试

要求：

- 覆盖真实 socket 连接
- 覆盖请求/响应链路
- 覆盖超时、断线、错误路径

### 4.3 新增功能必须带测试

规则：

- 协议改动必须补编解码测试
- Dispatcher 改动必须补分发测试
- 新消息号必须补至少一个集成测试
- 修 bug 必须带回归测试

## 5. 日志规范

### 5.1 日志级别

- `TRACE`
  非常高频的调试细节
- `DEBUG`
  调试定位信息
- `INFO`
  关键流程节点
- `WARN`
  可恢复异常、限流、非法请求
- `ERROR`
  明确失败、连接异常、业务错误
- `CRITICAL`
  严重不可恢复错误

### 5.2 日志内容要求

日志中尽量带上：

- 会话标识
- 远端地址
- 消息号
- 请求序号
- 关键状态值

避免：

- 大量无上下文日志
- 业务层重复打印相同错误
- 高频路径无节制 `INFO`

## 6. 协议规范

### 6.1 当前约定

当前协议：

```text
[4字节长度][2字节消息号][消息体]
```

### 6.2 未来建议

演进为：

```text
[4字节总长度]
[2字节消息号]
[4字节请求序号]
[1字节标记位]
[消息体]
```

### 6.3 消息号管理

- 按模块分段
- 禁止随意复用消息号
- 协议文档必须同步更新

推荐分段：

- `1-999`：系统消息
- `1000-1999`：网关类
- `2000-2999`：登录类
- `3000-3999`：房间类
- `4000-4999`：战斗类

## 7. 性能规范

### 7.1 不允许的做法

- 在网络线程里执行重 CPU 逻辑
- 在 handler 里做阻塞 I/O
- 无限制增长的发送队列
- 没有上限的大包处理

### 7.2 必须优先考虑的点

- Session 生命周期安全
- 线程切换边界清晰
- 广播与批量发包成本
- 慢连接和恶意连接处理

## 8. 配置规范

未来应补充统一配置模块，至少包含：

- 监听端口
- 线程池大小
- 最大包长
- 心跳间隔
- 心跳超时
- 发包限流阈值
- 日志级别

## 9. 文档维护规则

- 新模块落地时同步补 `docs/`
- 重要架构改动必须更新规划文档
- 每个阶段结束更新开发日志
- 每次发布前整理阶段变更摘要
