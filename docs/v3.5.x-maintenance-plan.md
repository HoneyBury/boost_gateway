# v3.5.x Maintenance Plan

更新时间：2026-07-17

本文档定义 `v3.5.0` 发布后的补丁版本边界。当前实现事实仍以
`docs/current-state.md` 为准，生产候选证据仍以对应候选 SHA 的 workflow
summary 和 artifact 为准。本维护线只接受兼容修复、发布工程、文档事实、
运行稳定性和验证增强，不在 patch 版本中扩大默认协议或业务能力。

## v3.5.1：发布与事实治理

目标：修复不改变运行时协议和默认行为的发布缺陷。

必须完成：

1. GitHub 仓库首页显示根 `README.md`，CI/CD 文档使用 `.github/CI-CD.md`。
2. Release workflow 从 `CHANGELOG.md` 提取当前版本正文，禁止只生成 compare 链接。
3. `CHANGELOG.md` 使用 `Unreleased`/当前 patch 版本边界，已发布版本不再吸收 tag 后变更。
4. 增加 MIT `LICENSE`，并将其纳入安装包和发行包门禁。
5. SDK native `4.1.0`、Python 和 C# 包元数据保持一致；兼容矩阵覆盖 Gateway `v3.5.x`。
6. `.tar.gz` 必须是真实 gzip 压缩，解压后以单一 `boost-gateway-vX.Y.Z-linux-x64/` 为顶层，不携带 `dist/` 前缀。
7. `current-state.md`、性能事实和 runner inventory 回填 `v3.5.0` 最终证据。
8. 为版本、Release notes、LICENSE、SDK 元数据和 archive layout 增加自动门禁。

冻结条件：本地治理门禁、Release package workflow、外部 SDK package consumer 和
全新下载校验均通过；候选运行时代码若未变化，可沿用 `v3.5.0` 的高成本风险结论，
但新候选仍必须生成自身的 bounded/R0/package provenance，不得伪造跨 SHA 的 R2/R3。

## v3.5.2：发行包与真实控制面

目标：提升发布包可消费性并补齐 `v3.5.0` 未阻断的真实环境专项。

- 在干净 Ubuntu 容器中执行解压、动态依赖、最小启动和 CMake consumer smoke。
- 生成 SBOM 和 GitHub artifact attestation；评估独立 debug-symbol 包。
- 明确 Python/C# wrapper 是源码示例还是正式发行资产；正式发行前增加 wheel/NuGet 安装验证。
- 为 AOI 或独立 runner 安装 Docker、kind、kubectl，执行 Operator status/delete 与 rollout/rollback E2E。
- 恢复至少一个备用 Linux runner，并按当前 Conan/Docker namespace 完成 G0-G5 准入。

已完成的阶段证据：

- Release run `29560450740`（候选 `9945028`）通过 clean Ubuntu package consumer 与 SPDX SBOM，artifact `8399167635`。
- Specialized E2E run `29563770679`（候选 `21a4815`）使用 kind `v0.32.0`、kubectl `v1.36.1` 与固定 node digest，完成 Ready、scale、rollout restart/undo、Operator restart、delete 和 cleanup；artifact `8400330394` 的 summary 为 `overall_pass=true`。
- 两个 run 绑定不同阶段提交，只证明对应能力已跑通；冻结前仍须在最终候选上刷新 Release/R0，正式 tag 验证 attestation，并按 GitHub API 记录备用 runner 的实际状态。

冻结条件：发行包 clean-environment smoke 通过，Operator kind 结果有当前候选 provenance，
runner inventory 与 GitHub API 一致。

## v3.5.3：长稳与性能事实

目标：补充不阻断 `v3.5.0`、但影响更高风险部署声明的长期证据。

- 完成 8h overnight soak，并保留失败轮、确认复测和宿主资源快照。
- 测定 1/2 核扩展、matchmaking/leaderboard、TLS 和 OTel 开销。
- 执行跨进程/真实网络 Redis/Raft 扰动与恢复测试。
- 对 Prometheus P99 和业务成功率告警进行多轮触发/恢复验证。
- 将 Windows 历史性能数据迁入 archive，当前性能正文以 Linux fixed-runner 为主。

冻结条件：8h、专项性能和告警验证均绑定同一候选；不能用推算值或历史平台数据替代。

## 下一次 Minor

以下事项不进入 `v3.5.x`：

- 将 generated protobuf/gRPC 升格为默认生产传输。
- 迁移内部 Raft raw JSON RPC 或进行破坏性 wire/API 变更。
- 正式新增跨语言 SDK 分发承诺、macOS ARM64 发布目标或新的业务功能面。
- 内建 JWKS/多 `kid`、持久化身份系统和可撤销 refresh token。

这些事项需要独立 ADR、兼容策略、迁移窗口和 minor 版本冻结，建议从 `v3.6.0` 开始评估。

## 维护分支与 GitHub 治理

- 使用 `release/3.5` 作为补丁维护线，候选从该分支的完整 SHA 触发。
- 每个 patch 使用独立 annotated tag；不得移动已发布 tag。
- `main` 和 `release/3.5` 应启用防强推与必需检查，`v*` tag 应由 ruleset 防删除/更新。
- 发布完成后删除同版本的临时 freeze/ops 分支，长期事实保留在 tag、Release、文档和 artifact。
- 每个版本建立 milestone/issues；关闭条件必须引用提交、workflow run 或文档事实。
