# 高性能游戏服务器框架规划

## 1. 项目目标

当前项目已经具备这些基础能力：

- Boost.Asio 网络异步框架
- 长度头协议、消息号、消息分发器
- 心跳检测、超时断开、发包限流、最大包长校验
- 基础单元测试和集成测试
- 按模块拆分的工程目录结构

如果要把它演进成一个“高性能游戏服务器框架”，目标不应该只是“能收发包”，而应该明确到以下能力：

### 1.1 网络层目标

- 支持高并发长连接
- 支持网关/逻辑服拆分
- 支持连接生命周期管理、限流、熔断、黑名单
- 支持协议编解码、压缩、加密、版本兼容
- 支持定时任务、心跳、踢线、广播

### 1.2 业务层目标

- 登录服
- 网关服
- 房间服
- 战斗服
- 匹配服
- 大厅服
- 后台 GM / 运维控制接口

### 1.3 工程层目标

- 清晰的模块边界
- 可扩展的消息注册与分发机制
- 可压测、可观测、可追踪、可灰度
- 可测试、可部署、可回滚
- 明确的开发规范、日志规范、Git 规范

## 2. 推荐总体架构

建议从“一体化单进程服务器骨架”逐步演进到“多服务可拆分架构”。

### 2.1 第一阶段：单进程多模块

适合当前阶段，重点是把抽象做对。

- `net`
  负责连接、协议、编解码、定时器、限流、线程模型
- `gateway`
  负责客户端接入、鉴权前置、路由转发、会话状态
- `login`
  负责登录、认证、账号校验、令牌签发
- `room`
  负责房间创建、加入、退出、房间广播
- `battle`
  负责战斗内消息、帧同步、状态广播、战斗结算

### 2.2 第二阶段：网关与逻辑服拆分

拆分为：

- `gateway_server`
- `login_server`
- `room_server`
- `battle_server`

此阶段要解决：

- 跨服消息协议
- 服务发现
- 内网 RPC 或消息总线
- 用户路由和会话迁移

### 2.3 第三阶段：高可用与高性能

- 多实例部署
- 服务注册与发现
- 配置中心
- 限流熔断
- 灰度发布
- 无状态化或弱状态化设计

## 3. 当前项目需要补齐的关键内容

### 3.1 协议系统

当前已有：

- `[4字节长度][2字节消息号][消息体]`

还需要补齐：

- 协议版本号
- 请求/响应序号
- 错误码体系
- 压缩标记
- 加密标记
- 序列化方案
  推荐选型：
  - protobuf：通用、成熟、跨语言
  - flatbuffers：读性能好，适合特定高频场景

建议目标包结构：

```text
[4字节总长度]
[2字节消息号]
[4字节请求序号]
[1字节标记位]
[N字节消息体]
```

### 3.2 网络线程模型

当前已有：

- `io_context` 网络线程
- `thread_pool` 业务线程池

还需要补齐：

- 连接绑定 executor 策略
- 广播与批量发包策略
- 慢连接检测
- 写队列积压监控
- 分服内连接管理器

### 3.3 会话管理

当前已有：

- `Session`
- 心跳超时
- 发包限流

还需要补齐：

- `SessionManager`
- 用户 ID 与 Session 映射
- 账号登录态与重连恢复
- 会话迁移
- 踢线策略
- 封禁策略

### 3.4 消息分发体系

当前已有：

- `MessageDispatcher`
- 消息号注册

还需要补齐：

- 模块级 handler 自动注册
- 消息上下文对象
- 鉴权前/鉴权后 handler 权限隔离
- 中间件机制
  例如：
  - 登录校验
  - 频控校验
  - 参数校验
  - 审计日志

### 3.5 登录系统

需要补齐：

- 登录请求/响应协议
- token 验证
- 重复登录处理
- 单端登录 / 多端登录策略
- 游客账号 / 正式账号支持
- 登录限频

### 3.6 网关系统

需要补齐：

- 连接准入控制
- 登录前消息白名单
- 登录后消息路由
- 网关层限流
- 断线重连

### 3.7 房间系统

需要补齐：

- 房间创建/加入/退出
- 房主机制
- 房间广播
- 玩家准备状态
- 房间生命周期管理

### 3.8 战斗系统

需要补齐：

- 战斗启动
- 玩家状态同步
- 帧同步/状态同步方案选择
- 战斗事件广播
- 战斗结束结算
- 战斗回放数据记录

## 4. 性能方向的具体建设项

### 4.1 内存与对象管理

- 自定义 Buffer 池
- 消息对象池
- 减少频繁 `std::string` 拷贝
- 大包零拷贝优化
- Session 级别复用缓冲区

### 4.2 并发与锁优化

- 避免全局大锁
- 将共享状态尽量局部化到 Session / Room / Battle 实例
- 广播路径减少锁竞争
- 业务线程池按服务类型拆分

### 4.3 协议与序列化

- 高频消息使用二进制协议
- 区分控制消息和大包消息
- 对广播消息考虑批处理

### 4.4 数据落盘与异步化

- 日志异步写
- 数据库写入异步化
- 战斗回放异步落盘
- 指标统计异步上报

## 5. 可观测性建设

高性能服务器不能只看“能跑”，还要能定位问题。

必须补齐：

- 结构化日志
- 请求链路追踪 ID
- 核心指标埋点
  - 在线人数
  - 每秒收包数
  - 每秒发包数
  - 平均包大小
  - 发包队列积压
  - 心跳超时数
  - 慢消息处理数
- Prometheus 指标或等价方案
- 崩溃现场保留

## 6. 测试体系规划

### 6.1 单元测试

覆盖：

- 协议编解码
- Dispatcher 注册与分发
- 业务服务注册
- 限流和边界值

### 6.2 集成测试

覆盖：

- 连接建立
- Echo
- 心跳
- 登录请求响应
- 断线超时
- 错误消息号处理

### 6.3 压力测试

必须新增：

- 多连接压测工具
- 登录风暴压测
- 广播压测
- 慢客户端压测
- 大包/恶意包压测

## 7. 分阶段开发路线

### 阶段 A：网络层稳定化

- 完成 `SessionManager`
- 完成消息中间件机制
- 完成错误码体系
- 完成更多网络层测试

### 阶段 B：业务骨架成型

- 完成登录模块
- 完成网关登录前白名单
- 完成房间模块
- 完成战斗模块最小闭环

### 阶段 C：工程化增强

- 指标采集
- 配置系统
- 压测工具
- Docker/部署脚本
- CI 扩展

### 阶段 D：性能优化与拆服

- 多进程/多服务拆分
- 内部通信协议
- 路由与服务注册
- 热点路径优化
