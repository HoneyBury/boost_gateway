# v3.3.2 P1 性能稳定化记录

日期：2026-05-17

提交基线：`63668b0e863f8b52996937c2ab46ddbc6d02bb74`

本记录用于归档 P0/P1 性能事实管线和低风险优化实验。当前阶段不扩展业务功能，优先保证 baseline 可解释、可重复、低波动。

## 交付结论

P1 默认稳定线保持 `backend_pool_size=1`。该配置下 Release baseline 单轮通过，且所有 case 的 rejected/failed/forced_timeout 均为 0。

多 backend connection pool 仍作为实验项保留在采集参数中，不进入默认 release baseline。显式 `backend_pool_size=8` 的 smoke 实验在 battle 场景触发 backend_error/rejected，说明当前 backend connection 生命周期、熔断和多连接池交互仍会放大波动。

## 实验风险记录

报告路径：

- `runtime/perf/p1-backend-pool8-smoke-experiment/report.md`
- `runtime/perf/p1-backend-pool8-smoke-experiment/summary.json`

命令：

```bash
python3 scripts/collect_v2_perf_baseline.py \
  --build-dir build/release \
  --run-preset smoke \
  --repetitions 1 \
  --backend-pool-size 8 \
  --output-root runtime/perf/p1-backend-pool8-smoke-experiment
```

结果：

| Case | 结果 | 说明 |
| --- | --- | --- |
| echo-20-10s | passed | p99 2ms，rejected/failed/forced_timeout 为 0 |
| battle-2-10s | failed | rejected=2，body=backend_error |

结论：`backend_pool_size>1` 不作为当前默认优化项。后续若继续推进，需要先设计 backend connection 生命周期、连接健康检查、熔断粒度和连接池回收策略，再进入性能优化。

## P1 代码与报告能力

- `scripts/collect_v2_perf_baseline.py`：生成 `resource_analysis` 和 `report.md`，并记录 `backend_connection_pool_size`。
- `scripts/collect_release_baseline.py`：release summary 记录 `performance_summary_path` 与 `performance_report_path`。
- `docs/performance-baseline.md`：补充报告结构、资源字段和 P1 实验口径。

## 后续约束

- 默认 release baseline 必须保持 `backend_pool_size=1`，除非新的专项报告证明更大连接池稳定。
- `battle-100` p99 当前仍为 200ms，虽然通过 250ms gate，但接近高并发 battle 的重点观察区。
- 后续优化必须带同机同命令对比报告，避免凭单次 smoke 结论改变默认路径。
