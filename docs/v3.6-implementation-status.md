## 生产 workflow 三平台路由

默认分支已让 Release、R0、performance、long soak /
capacity、P5/P6 diagnostics、readiness 和 published asset verification 显式选择
`linux-x64`、`linux-arm64` 或 `macos-arm64`。共享 composite action 对真实 host
OS/CPU、Conan profile/lockfile、持久 cache、build directory、Docker platform 和
artifact suffix 做 fail-closed 绑定。Release tag 生成三平台独立 archive/SBOM；Linux
consumer 校验 ELF/OCI architecture，Mac consumer 校验 Mach-O/C ABI/native CMake。
路由和预冻结 ARM workflow 已验证；最终 exact-SHA evidence 必须在本次版本冻结后刷新。

## 激活顺序

1. 将项目版本和当前事实源冻结为 `v3.6.0` 候选，冻结后不再混入功能或证据脚本变更。
2. 在唯一冻结 SHA 上按 `linux-x64`、`linux-arm64`、`macos-arm64` 刷新 Release/R0/R5、
   JWKS、SDK、符号、baseline 和 required stability/capacity evidence。
3. 补齐 macOS 签名/notarization 与正式发布凭据；未具备前不得创建 tag 或声明正式资产。
4. 冻结候选 tag 后发布 GitHub Release 资产，再由独立 workflow 下载并核对
   checksum、SBOM、provenance、包消费和符号映射。
5. 只有具体版本/RID 的全部证据闭环后才更新 compatibility matrix；PyPI、
   NuGet.org、Raft protobuf writer 默认启用仍需独立激活决策。
