# ADR: v3.6 独立调试符号资产

- 决策 ID：`v36-debug-symbols`
- Status：`accepted_for_implementation`
- Target Minor：`v3.6.0`
- Default Activation：`blocked_until_gates_pass`
- 日期：2026-07-21

## 背景

当前 release 只构建 `Release`，没有 `RelWithDebInfo`、符号拆分、build-id 映射、
独立 debug archive 或 symbol server 契约。已发布 `v3.5.3` 资产不可重建或补发
同名符号包。

符号资产必须与运行时二进制逐字节身份绑定；“存在 `.debug` 文件”不能证明它
能够解析线上栈，也不能让未 strip 的大二进制替代正式 runtime package。

## Decision / 决策

- 首个支持目标是 Linux x64 ELF。release 候选使用同一次 `RelWithDebInfo` 构建
  生成 runtime 与 symbols：runtime binary 执行 `objcopy --only-keep-debug`、
  strip 和 `--add-gnu-debuglink`，symbols archive 保存对应 `.debug` 文件。
- 每个二进制记录 build-id、runtime SHA-256、debug file SHA-256、debuglink 和
  相对路径。门禁必须用已知崩溃/地址验证 symbolizer 能还原函数和源代码行。
- runtime tarball 名称和安装布局保持兼容；symbols 使用独立
  `boost-gateway-<version>-linux-x64-debug-symbols.tar.gz`，不会被普通安装命令
  自动展开到生产节点。
- runtime archive 与 symbols archive 分别进入 checksum、provenance、SBOM 和
  published-asset verification。符号 SBOM 必须覆盖实际文件 hash，并通过映射
  manifest 绑定 runtime asset，不复用一个无法区分 subject 的 attestation。
- 首批不建设 symbol server。运维 runbook 说明离线下载、验签、按 build-id
  定位和栈回溯流程。

## Compatibility / 兼容策略

- 符号资产是 additive，不改变 C++ API、C ABI、wire protocol、默认配置或
  runtime 文件名。
- `RelWithDebInfo` 与既有 Release 优化/性能必须做 bounded 对照；若编译选项或
  strip 改变运行行为，release 阻断，不能通过放宽性能门槛发布。
- `v3.5.x` 不补发或重建 symbols。`v3.6` 每个不可变 tag 只接受同一构建产生的
  一组 runtime/symbol assets。
- macOS dSYM 是平台后续扩展，只有 macOS ARM64 发布资产稳定且另行补齐 UUID、
  codesign/notarization 边界后才能加入；Linux symbols 不依赖它。

## Migration Window / 迁移窗口

1. 在非 tag 候选上生成 runtime/symbol pair，验证 strip/debuglink/build-id 和
   stack resolution，不上传 Release。
2. 将两个资产及各自 SBOM/attestation 纳入 release manifest 和独立线上复验。
3. 一个 `v3.6.x` 候选完成同 SHA 复验后，symbols 才成为 release 必需资产。
4. 如符号步骤失败，在形成发布承诺前可回退为无独立 symbols 的 candidate；
   一旦 release manifest 声明 symbols，缺失必须阻断整个 tag 发布。

## Release Assets / 发布资产

- Linux x64 stripped runtime tarball。
- Linux x64 debug-symbols tarball 和 build-id 映射 manifest。
- 两个 archive 的独立 SPDX SBOM、SHA-256、provenance/SBOM attestation。
- 离线符号化 runbook 和 published-asset verification summary。

## Validation Matrix / 验证矩阵

| 层级 | 必须覆盖 |
|---|---|
| Build | `RelWithDebInfo`、build-id 唯一且稳定、runtime 已 strip、debug file 保留 DWARF、debuglink 正确 |
| Mapping | 每个发布 ELF 恰有一个符号文件；manifest hash/build-id 与 runtime 和 debug archive 一致 |
| Symbolization | 已知函数地址和受控崩溃能恢复函数/文件/行；错误 symbols 明确失败而非给出误导结果 |
| Runtime | clean Ubuntu consumer、完整 tests/gates、启动/full-flow 与当前 Release 基线无行为退化 |
| Supply chain | 两个资产 checksum、SBOM、provenance、subject binding 和独立线上下载复验 |

## Activation and Rollback / 激活与回滚门槛

只有 runtime tests、symbolization、性能对照和供应链矩阵在同一候选 SHA 全部通过，
才能把 symbols 加入必需 release manifest。不得用另一提交或本地重编译符号解析
已发布二进制。

资产错误时不覆盖 tag 或同名文件，发布修复版本并标记受影响 build-id。生产
runtime 可继续运行，但 release readiness 必须记录符号诊断缺口。

## Out of Scope / 不支持范围

- 不在首批实现中提供公共/私有 symbol server、自动 crash upload 或 core dump
  数据保留服务。
- 不承诺 macOS dSYM、Windows PDB、源码打包或第三方依赖完整调试符号。
- 不修改或补发 `v3.5.3` 的任何资产。

## Manifest Contract

- Compatibility Window: symbols are additive beginning with v3.6; no v3.5.x asset is rebuilt or republished
- Migration Window: validate unpublished runtime-symbol pairs before adding both assets and their subject-bound attestations to the required release manifest
- Release Asset: Linux x64 stripped runtime archive
- Release Asset: Linux x64 debug-symbols archive and build-id manifest
- Release Asset: asset-specific SBOM, checksums and attestations
- Validation Gate: ELF strip, debuglink and build-id mapping
- Validation Gate: known-address and controlled-crash symbolization
- Validation Gate: runtime regression and independent supply-chain verification
