# ADR: v3.6 Python wheel 与 NuGet 正式分发

- 决策 ID：`v36-sdk-distribution`
- Status：`accepted_for_implementation`
- Target Minor：`v3.6.0`
- Default Activation：`blocked_until_gates_pass`
- 日期：2026-07-21

## 背景

当前正式 SDK 面是 C++ CMake package 和 C ABI shared library。Python/C# 已有
薄 wrapper、包元数据、native 版本检查和 full-flow 示例，但仍要求使用者自行
提供 native library，不构成经过平台矩阵、签名和 clean install 验证的 wheel
或 NuGet 分发。

Gateway `v3.5.x` 与 native SDK `4.1.0` 保持 `4.x` 主版本兼容。本决策不把
Gateway 与 SDK 版本号强制绑定，也不借打包变更扩大公共业务 API。

## Decision / 决策

- 下一次正式包版本从 SDK `4.2.0` 开始，避免覆盖已存在的 `4.1.0` 元数据。
  Gateway `v3.6.x` 继续接受兼容的 SDK `4.x`；破坏性 C/C++ ABI 或 wire 变化
  仍必须提升 SDK 主版本。
- Python 发布平台 wheel，包内携带匹配 RID 的 C ABI shared library 和许可证，
  wrapper 优先加载包内库，同时保留 `BOOST_GATEWAY_SDK_LIBRARY` 诊断覆盖入口。
  初始 Linux 目标为 pinned manylinux x86_64 构建环境；macOS ARM64 wheel 必须
  等 `v36-macos-arm64` 门禁通过后单独加入。
- NuGet 包 `BoostGateway.Sdk` 包含 managed wrapper、文档、许可证，以及
  `runtimes/<rid>/native/` 下的匹配 native library。首个正式 RID 是
  `linux-x64`；`osx-arm64` 只有在 macOS ADR 的 install/full-flow 验证通过后
  才能宣称支持。包不得静默下载运行时二进制。
- 第一阶段把 wheel、`.nupkg`、checksum、SBOM 和 provenance 作为 GitHub
  Release 资产，完成与当前 tarball 相同的独立下载验证。发布到 PyPI 或
  NuGet.org 属于第二阶段，必须使用 trusted publishing/专用发布身份、不可变
  版本和 registry 安装复验；没有凭据与防重发策略时不得宣称公共仓库可用。
- C++ package 与 C ABI 仍是 native 事实源。跨语言 wrapper 不复制协议状态机，
  不新增 demo 专属 API，也不内嵌独立版本的 native 实现。

## Compatibility / 兼容策略

- 现有源码 wrapper 与环境变量加载路径在 `v3.6.x` 保留，给源码构建和诊断
  使用；正式包只是新增可复现安装路径。
- 包导入/创建 client 前继续校验 native 主版本。managed/package version、
  native `gsdk_version()` 和构建 manifest 必须可对账。
- 不同 RID 的 native 文件不能放入错误平台包；不支持的平台必须安装失败或在
  创建 client 前给出明确诊断，不能退回系统中偶然存在的同名库。
- wheel/NuGet 卸载后不得遗留或删除用户自行安装的 native library。

## Migration Window / 迁移窗口

1. 先在 CI 生成未发布的 `4.2.0` 候选包，执行 clean install、SDK consumer 和
   真实 Gateway full-flow。
2. 在一个冻结候选 SHA 上发布 GitHub Release 资产并从线上资产独立复验。
3. 至少一个 `v3.6.x` 候选完成资产复验后，才可分别启用 PyPI/NuGet.org
   publishing。registry 发布失败不得影响 C++/C ABI 和 GitHub 资产的回滚。
4. 源码 wrapper 在整个 `v3.6.x` 保留；后续是否仅文档降级需另行决策。

## Release Assets / 发布资产

- `boost_gateway_sdk-4.2.0-<python>-<abi>-<platform>.whl`。
- `BoostGateway.Sdk.4.2.0.nupkg`，后续如需 symbols package 必须独立命名。
- 每种包的 SPDX SBOM、SHA-256、build provenance、包内容 manifest 和许可证。
- 兼容矩阵必须逐项列出 Gateway、SDK、Python、.NET、RID、native ABI 和状态。

## Validation Matrix / 验证矩阵

| 层级 | 必须覆盖 |
|---|---|
| Build | pinned toolchain、可复现包名/版本、包内容白名单、license/SBOM/checksum、禁止绝对构建路径 |
| Python | clean venv 安装/导入/卸载；包内 native 加载；错误 RID/版本诊断；login/room/battle/push/reconnect/heartbeat full-flow |
| .NET | clean project `dotnet add package`、RID native resolution、allocation/version failure、同一 full-flow 能力矩阵 |
| Native | CMake package consumer、C ABI tests、导出符号/ABI 基线，确保跨语言包不改变 native 契约 |
| Published asset | 从 GitHub Release 或 registry 独立下载，验证 checksum/provenance/SBOM 后在无源码目录运行 |
| Platform | `linux-x64` 必须通过；`osx-arm64` 只有 macOS ADR 门禁通过后才能加入 supported 列 |

## Activation and Rollback / 激活与回滚门槛

只有某个具体版本和 RID 的 build、clean install、full-flow、线上资产与兼容矩阵
全部通过，才能把该组合标为 supported。一个平台通过不能替代另一个平台，也
不能把本地 `setup.py`/`.csproj` 成功当作正式包证据。

包发布后不可覆盖同版本。发现问题时撤销推荐、发布修复版本并保留旧资产审计
事实；C++/C ABI 路径继续可用，不以删除历史包作为回滚手段。

## Out of Scope / 不支持范围

- 本决策不承诺 Windows RID、macOS x64、移动平台、静态链接 wheel 或 AOT。
- 不把 gRPC 实验 target 暴露为 Python/C# 默认 transport。
- 不提供业务 demo 专属 SDK API，也不承诺公共 registry 在首批实现中启用。

## Manifest Contract

- Compatibility Window: source wrappers and explicit native-library loading remain available throughout v3.6.x; SDK 4.x compatibility is retained
- Migration Window: validate SDK 4.2.0 candidate packages and GitHub Release assets before independently enabling immutable PyPI or NuGet.org publication
- Dependency: v36-macos-arm64
- Release Asset: platform Python wheels
- Release Asset: BoostGateway.Sdk NuGet package
- Release Asset: per-package SBOM, checksums, provenance and compatibility matrix
- Validation Gate: reproducible package build and content policy
- Validation Gate: clean Python and .NET install, uninstall and native resolution
- Validation Gate: real Gateway full-flow and independent published-asset verification
