# ADR: v3.6 Raft schema 与滚动迁移

- 决策 ID：`v36-raft-wire-schema`
- Status：`accepted_for_implementation`
- Target Minor：`v3.6.0`
- Default Activation：`blocked_until_gates_pass`
- 日期：2026-07-21

## 背景

Raft 当前存在三套不同的隐式 JSON 契约：RequestVote/AppendEntries RPC wire、
节点磁盘 state，以及 matchmaking/leaderboard state-machine command。RPC parser
对缺失字段使用零值；磁盘损坏或未来 schema 不能被当作全新节点继续启动；
未知 command 版本也不能在状态机未应用时仍推进 applied 事实。

本决策形成时，仓库尚无 `proto/v3/raft.proto`，默认 Conan 图不包含 protobuf，
protobuf/gRPC 绑定在实验选项中。因此 Raft wire 迁移首先需要把 protobuf runtime
从 gRPC transport 解耦，但这不改变 gRPC 的 deferred 状态。当前实现进度见下文。

## Decision / 决策

### RPC wire

- 新增内部 `proto/v3/raft.proto`，覆盖 RequestVote、RequestVoteReply、
  AppendEntries、AppendEntriesReply、LogEntry 和显式 protocol version。
- 默认构建仅引入 protobuf runtime，不引入或默认启用 gRPC。Raft RPC 继续走
 现有 BackendEnvelope/连接路径，payload 改为有版本的 deterministic protobuf。
- 新 codec 能识别 legacy JSON 与 protobuf，严格限制 payload/entry/command 大小、
  必需 ID、term/index 范围和未知版本。解析错误返回稳定协议错误，不把缺失字段
  默认为有效零值。
- 老节点不能读取 protobuf，因此新 writer 只有在所有 voting peers 明确通告
  protobuf capability 后才能启用。混合版本集群一律 old-write/new-read；全部
  升级后才由显式配置切为 new-write。不得用超时猜测 peer capability。

### 磁盘 state

- 先于 RPC 默认切换实现 versioned state codec。新 state 包含 schema version、
  node identity、term、voted_for、log/commit/applied 元数据和完整性校验，并继续
  使用临时文件、fsync/等价持久化与原子 rename。
- legacy v0 state 只读迁移一次，成功校验后写成 v1。损坏、类型错误、截断、
  identity 不匹配或未知未来版本都必须阻止节点启动，不得静默回到 term 0。

### State-machine command

- matchmaking 与 leaderboard command 使用独立 versioned protobuf/typed schema，
  command type 与字段均有界。legacy command 在兼容窗口内可读。
- 未知/不支持版本必须使 apply 明确失败、节点 unhealthy 并阻止对外提供一致性
  读写；不得把该 index 标记为已成功应用。失败状态和 index 必须可观测。

## Compatibility / 兼容策略

- `v3.6.x` 保留 legacy JSON reader，但新增功能不得扩展 legacy 字段。
- 混合集群阶段固定 old-write/new-read；protobuf writer 是部署显式开关，并由
  peer capability 与集群健康门禁共同约束。
- 磁盘 v0 到 v1 的迁移必须在原文件旁保留有界备份和迁移记录；回滚到只懂 v0
  的二进制前必须使用受支持的 downgrade 工具，禁止手工删 state。
- schema field number 永不复用。删除字段必须 reserved；未知 protobuf fields
  按 schema 兼容规则保留或忽略，但未知 protocol/command major version fail closed。

## Migration Window / 迁移窗口

1. Phase A：先交付严格 disk codec、legacy fixtures 和损坏/future-version
   fail-closed 测试，不改变 RPC writer。
2. Phase B：交付 protobuf schema/runtime、dual reader、peer capability 和 golden
   vectors；集群仍写 legacy JSON。
3. Phase C：所有 voting peers 升级并通过专项门禁后，显式启用 protobuf writer；
   至少一个 `v3.6.x` release 保留 legacy reader 和 writer 回滚开关。
4. legacy writer/reader 最早在 `v3.7.0` 由新 ADR 移除。本决策不自动删除它们。

## Implementation Progress / 实现进度

- Phase A 已完成：`RaftStateStore` 提供严格 v1 disk codec、canonical SHA-256、node identity 和 term/index/log 不变量校验；首个 legacy v0 转换保留固定 sidecar，后续往返使用内容寻址、带 generation 且每方向最多八对的有界历史。损坏、孤立或超限历史均在替换主状态前 fail closed。
- Raft 节点在恢复到损坏、截断、identity 不匹配或未来版本状态时同步失败；matchmaking/leaderboard 只有在 Raft 恢复与首次持久化成功后才开始监听。运行期持久化失败会锁存 unhealthy 并拒绝投票、复制与业务 append。
- Phase B 实现候选已落地：`proto/v3/raft.proto` 定义显式 protocol version 和共识/capability envelope；默认 Conan 图以 `with_raft_protobuf=True` 独立引入 protobuf/abseil，不启用 gRPC；codec 通过 `BGRT` framing 严格双读 legacy JSON/protobuf v1，确定性 RequestVote/AppendEntries golden vectors 已固化，peer capability 只由独立 protobuf 请求/响应确认，探测失败或 peer 回滚会撤销旧 capability 缓存。
- 三节点 protocol-profile E2E 已覆盖逐节点升级、leader 停止与重选、committed log catch-up 和逐节点回滚，并断言共识 writer 全程保持 legacy JSON。`raft_state_tool` 已实现严格 v1-to-v0 转换、原子替换、v1 精确备份和 checksum 审计记录；负向测试覆盖损坏、身份/sidecar 冲突和 legacy 状态被继续修改后的 fail-closed 行为。
- 真实进程门禁已使用 `v3.5.3` 提交 `b9c348b4...` 与当前候选的不同 backend binary 在本机完成十三阶段双周期升级/回滚，逐阶段断言三节点读回、提交索引和 `v0 -> v1 -> v0 -> v1 -> v0` 磁盘轨迹，第二次回滚必须使用独立 history sidecar。release workflow 已将该门禁与严格离线 Conan、specialized、data-recovery、package-consumer、SBOM 绑定，并要求预置 legacy 制品 SHA-256；exact candidate SHA 的 Linux fixed-runner run ID 尚未生成，本地 summary 不构成激活证据。
- Phase C 尚未实现：RequestVote/AppendEntries writer 仍固定使用 legacy JSON，capability 不完整不会推断支持或切换 writer；state-machine command codec 和 fixed-runner 发布资产证据仍待交付。
- 因此本 ADR 继续保持 `accepted_for_implementation` / `blocked_until_gates_pass`；Phase A/B 实现完成不代表 Raft wire/command 迁移已激活。

## Release Assets / 发布资产

- `proto/v3/raft.proto`、生成代码版本信息和 wire/disk/command golden fixtures。
- 默认 Conan lockfile 中独立 protobuf runtime、fixed-runner 离线 cache identity、
  更新后的 SPDX dependency/SBOM 与 clean package consumer 证据。
- `docs/deployment/raft-schema-migration-runbook.md`、state 备份/downgrade 工具、混合版本集群 summary。
- Raft schema 是内部契约，不因此承诺公共 gRPC endpoint 或 SDK API。

## Validation Matrix / 验证矩阵

| 层级 | 必须覆盖 |
|---|---|
| Codec | legacy/protobuf golden round-trip；malformed/truncated/oversized；missing ID；unknown version；term/index 边界；deterministic bytes |
| Disk | v0->v1；原子写；损坏/checksum/type/identity/future version 启动失败；备份与受支持 downgrade |
| Command | matchmaking/leaderboard legacy+new command；unknown version 不推进 applied；失败健康状态和恢复 |
| Mixed cluster | 3 节点逐个升级和逐个回滚；old-write/new-read；capability 不完整拒绝 new writer；leader 切换期间不丢 committed entry |
| Recovery | follower catch-up、leader restart、committed replay、持久化恢复、网络扰动和跨进程真实 RPC |
| Release | protobuf 默认依赖的严格离线 Conan install、完整测试、SBOM、package consumer，不启用 gRPC transport |

## Activation and Rollback / 激活与回滚门槛

disk codec 可在其 fail-closed 和迁移测试完成后先落地。protobuf RPC writer 只有在
同一候选 SHA 的 codec、混合集群、恢复、离线依赖和发布资产门禁全部通过后才能
显式启用，且不能自动成为所有部署的默认值。

回滚优先切回 legacy writer，新 reader 保持；涉及已迁移磁盘 state 时必须先用
受支持 downgrade 工具。任何无法识别的 state/command 都必须停止并告警，不能
通过删除文件或跳过 log entry 恢复服务。

## Out of Scope / 不支持范围

- 不把 gRPC 升格为 Raft 或客户端默认 transport。
- 不在本决策中改变 Raft 算法、quorum、成员变更模型或一致性等级。
- 不承诺 legacy JSON 在 `v3.6.x` 内删除，也不允许 big-bang 全集群切换。

## Manifest Contract

- Compatibility Window: v3.6.x keeps legacy JSON readers and at least one release keeps the legacy writer rollback path; removal requires a new v3.7+ ADR
- Migration Window: disk codec first, then dual-read legacy-write, followed by an explicitly gated protobuf writer while legacy rollback remains available
- Release Asset: proto/v3/raft.proto and golden fixtures
- Release Asset: protobuf runtime lockfile and SPDX dependency evidence
- Release Asset: mixed-version cluster and state migration summaries
- Validation Gate: strict wire, disk-state and command codec tests
- Validation Gate: three-node mixed-version rolling upgrade and rollback
- Validation Gate: Raft recovery and strict offline release dependency gates
