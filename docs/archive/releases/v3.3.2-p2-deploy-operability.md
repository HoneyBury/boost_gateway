# v3.3.2 P2 部署运维收束记录

日期：2026-05-17

提交基线：`b9705de`

本记录归档 P2 阶段的部署可运行性、运维配置一致性和防回归门禁。当前阶段不扩展业务功能，目标是让 Docker Compose、systemd、Kubernetes、Prometheus 和文档口径与真实进程行为一致。

## 交付结论

P2 已完成部署入口收束：

- Docker Compose gateway 显式连接 5 个后端服务名，不依赖容器内 `127.0.0.1` 默认值。
- Docker Compose 与 systemd gateway 启动依赖覆盖 5 个后端服务。
- 后端容器、Kubernetes 后端探针均使用 TCP 端口探测，不再假设后端暴露 HTTP `/health`。
- Prometheus 只 scrape 当前真实存在的 gateway HTTP `/metrics`。
- systemd 覆盖 gateway、login、room、battle、match、leaderboard 6 个核心进程。
- match/leaderboard 示例进程改为信号驱动常驻，适合 Docker/systemd 非交互环境。
- `scripts/check_deploy_operability.py` 成为 P2 部署运维门禁，覆盖 Dockerfile、两套 Compose、systemd、K8s、Prometheus、入口参数和可选构建产物。

## 关键修复

| 区域 | 修复 |
| --- | --- |
| Dockerfile | backend runtime 安装 `netcat-openbsd`，健康检查使用 `nc -z` |
| Compose | gateway 显式配置并等待 login/room/battle/matchmaking/leaderboard 后端 |
| Compose | leaderboard 默认连接 Compose Redis，并等待 Redis healthy |
| systemd | 新增 `boost-match-backend.service`、`boost-leaderboard-backend.service` 并纳入 CMake install |
| K8s | 5 个 backend deployment 的 liveness/readiness 改为 `tcpSocket` |
| Monitoring | `env/monitoring/prometheus.yml` 移除后端 HTTP scrape，仅保留 gateway `/metrics` |
| Runtime entry | `v2_gateway_demo` 解析 `--matchmaking-host/port` 与 `--leaderboard-host/port` |
| Docs | `deploy/README.md`、`env/README.md` 对齐当前 6 服务拓扑与健康检查语义 |

## 当前限制

- 后端 HTTP `/health`、`/metrics` 尚未标准化；当前部署面明确按 TCP 存活探测处理。
- Prometheus 后端指标仍需后续把文件指标、sidecar 或后端 HTTP 管理面标准化后再接入。
- K8s manifests 已对齐探针语义，但本阶段只做静态配置验证；真实 kind/集群 rollout 可作为后续固定 runner 验证项。
